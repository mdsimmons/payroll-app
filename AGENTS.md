# Payroll App - Agent Guide

## Quick Start

```bash
pip install flask
python3 app.py
```

Then open http://localhost:5000

**macOS note:** If `/usr/bin/python3` fails with Xcode license error, run `sudo xcodebuild -license accept` first.

## Deploy to Render

1. Push this repo to GitHub
2. In Render dashboard, create a **New Web Service** and connect your repo
3. Render auto-detects `render.yaml` — or manually set:
   - **Build Command:** `pip install -r requirements.txt`
   - **Start Command:** `gunicorn app:app`
4. Set environment variables in Render dashboard:
   - `SECRET_KEY` — auto-generated by render.yaml
   - `ADMIN_USER` — default `admin`
   - `ADMIN_PASS` — default `admin`
5. Deploy — your app will be live at `https://payroll-app.onrender.com`

## Architecture

```
payroll_app/
  app.py              # Flask web server (port 5000) - main entrypoint
  main.py             # CLI version (legacy, superseded by web UI)
  payroll.py          # Core calculation engine
  tax_tables.py       # 2026 tax brackets, standard deductions, FICA rates
  employees.json      # Employee records (read/written by API)
  payroll_history.json# Payroll run history (auto-created on first run)
  templates/index.html# Single-page web UI
  static/             # Static assets (currently empty)
```

## Key Conventions

### Tax Data
- `tax_tables.py` contains **2026** values only. When tax years change, all brackets/deductions/FICA limits must be updated.
- CA and NY use filing-status-specific bracket keys: `brackets_single`, `brackets_married`, `brackets_head_of_household`
- Flat-tax states (TX, FL, WA, NC) use a generic `brackets` key with a single rate
- `calculate_state_tax()` in `payroll.py` resolves brackets by trying `brackets_{filing_status}` first, falling back to `brackets`
- When `tax_settings.json` exists, state tax uses a **wage-bracket lookup** — weekly gross pay is matched against `[min, max, amt_0, amt_1, amt_2]` rows and the allowance column picks the dollar amount. No annualization or percentage calculation.

### Employee Model
- Employee records live in `employees.json` as `{"employees": [...]}`
- Fields: `id`, `name`, `hourly_rate`, `filing_status`, `federal_allowances`, `state`, `state_allowances`
- Employee model does **not** include health insurance or 401k fields (user requested removal)
- `calculate_pre_tax_deductions()` still supports `pre_tax_deductions` dict for future use

### Payroll Calculation
- Weekly pay periods, 26 periods/year assumed
- Gross pay: overtime at 1.5x after 40 hours
- Federal tax: annualizes per-period income, subtracts allowances + standard deduction, applies brackets, then divides back
- State tax: same annualization approach
- FICA: SS (6.2%) capped at $184,500, Medicare (1.45%) uncapped, additional Medicare (0.9%) above $200k

### API Endpoints
- `GET /api/employees` - list all
- `POST /api/employees` - add new (auto-generates EMP### id). Supports `pin` field for time clock.
- `PUT /api/employees/<id>` - update
- `DELETE /api/employees/<id>` - remove
- `POST /api/payroll` - run payroll with `{hours, week_info}` payload; saves to history
- `GET /api/history/weeks` - list all saved pay weeks
- `GET /api/history/<week_key>` - get full payroll for a week
- `DELETE /api/history/<week_key>` - remove a week from history
- `GET /api/schedules` - get all employee schedules
- `PUT /api/schedules` - save all employee schedules
- `GET /api/timeclock/now` - get current clock status (public, for tablet)
- `POST /api/timeclock/clock` - clock in/out with `{employee_id, pin}` (public)
- `GET /api/timeclock/history` - get all clock entries (admin)
- `POST /api/timeclock/delete` - delete a clock entry by index
- `GET /api/timeoff` - get all time off requests (admin)
- `POST /api/timeoff/request` - submit time off request (public, uses session or body)
- `POST /api/timeoff/respond` - approve/deny a request (admin)
- `POST /api/employee/login` - employee login with `{employee_id, pin}`
- `GET /api/employee/session` - check employee session
- `POST /api/employee/logout` - employee logout
- `GET /api/employee/paystubs` - get employee's paystubs
- `GET /api/employee/schedule` - get employee's schedule (includes `status` field: draft/published). Accepts optional `week_start` (YYYY-MM-DD) query param; if absent or matching current schedule, returns that schedule; otherwise returns empty shifts.
- `GET /api/employee/timeoff` - get employee's time off requests
- `POST /api/schedules/publish` - publish current schedule
- `POST /api/schedules/unpublish` - unpublish current schedule  
- `POST /api/schedules/copy` - copy previous week's schedule (`{source_week}`)
- `GET /api/notifications` - list notifications (admin sees all with is_read, employee sees theirs with is_read)
- `POST /api/notifications/read` - mark notification as read with `{id}`. Admin marks with "admin", employee marks with their employee_id.

### Login / Authentication
- All routes except `/api/login`, `/api/logout`, `/api/session`, and `/` require authentication via `@require_login` decorator
- Default credentials: `admin`/`admin` (overridable via `ADMIN_USER`/`ADMIN_PASS` env vars)
- Credentials can also be customized by creating/edit `users.json` (flat `{username: password}` dict)
- Flask sessions with a random secret key (overridable via `SECRET_KEY` env var)
- Frontend checks `/api/session` on load; shows login page if no session
- `POST /api/change-password` — updates username/password. Requires `current_password`, `new_password`, optional `new_username`. Updates `users.json` and the session.

### History Storage
- `payroll_history.json` is a flat dict keyed by `week_label` string
- No database - all state is JSON files

### New Data Files
- `schedules.json` - employee shift schedules (`{"shifts": {"EMP001": [{"start": "09:00", "end": "17:00", "notes": ""}, ...]}, "status": "draft", "open_shifts": [], "week_start": "", "week_label": ""}`)
- `timeclock.json` - clock in/out entries (`{"entries": [{"employee_id", "date", "clock_in", "clock_out", "hours", "status"}]}`)
- `time_off.json` - time off requests (`{"requests": [{"id", "employee_id", "start_date", "end_date", "reason", "status"}]}`)
- `users.json` - admin login credentials (auto-created when changed via Settings)
- `notifications.json` - in-app notifications with read tracking (auto-created)
